A practice romhack of The Legend of Zelda: A Link to the Past (JP 1.0)

Explanation of DG watch values

Row 1

Coordinates
Displays your current X (white) and Y (yellow) coordinates, with the full 16-bit precision
Coordinate sync
Indicates the sync status of your coordinates, based on whether your actual room ID matches your calculated room ID.
Actual room ID
Displays your current, actual room ID; i.e. the value at address $A0(2 bytes)
Corrected room ID
Displays your corrected room ID, calculated from your current coordinates. When this value is in sync with your actual room ID, it will be gray. When out of sync, it will be red.
Quadrant
Displays your actual quadrant, based on the values at addresses $A9 and $AA. The calculation is a simple table, shown below:
$A9 $AA
$00 $02
$00 Upper Left Upper Right
$02 Bottom Left Bottom Right
Correct quadrant
Displays your corrected quadrant, calculated from your current coordinates. When this is in sync with your actual room ID, the icon will be gray. When out of sync, it will be red.

Row 2

Room flags
Displays the current room flags, based on the values at addresses $0400, $0401, $0403, and $0408.
Icon Address Bit Icon Address Bit
Boss heart collected $0403 7 Door 1 $0401 7
Key flag 1 $0403 6 Door 2 $0401 6
Key/Misc flag 2 $0403 5 Door 3 $0401 5
Chest 5 $0403 4 Door 4 $0401 4
Chest 4 $0403 3 Upper left quadrant $0408 3
Chest 3 $0403 2 Upper right quadrant $0408 2
Chest 2 $0403 1 Lower left quadrant $0408 1
Chest 1 $0403 0 Bottom right quadrant $0408 0
Overlay buffer

This is the location in memory that determines where the buffer is that causes the major effects of overlay corruption. First, where to find the pointer is determined by the value at address $BA(2 bytes).

When $BA is $0000, the buffer's pointer is read from ROM, by looking at the value of address $04BA(2 bytes), multiplying it by 3, and using it as an offset into a table beginning at $04:E9A1. When this is the case, the address displayed will be preceded by the "ROM pointer" icon .

When $BA is any non-zero value, the buffer's pointer is read from WRAM, by looking at the values in addresses $B9 (bank byte) and $B7 and $B8 (16-bit address). When this is the case, the address displayed will be preceded by the "RAM pointer" icon .

While knowing exactly what "data" (it may not actually be data) will be read from the buffer is not important, knowing the effects of each location is helpful for roughly determining the effects of the corruption. It can also be used to group rooms together, as rooms with the same pointer will likely have the same results.

Buffer results warning

If there as a potential problem with the current buffer pointer, an icon will be displayed after it.

Vanilla warning
This icon will display if the buffer is pointing to a ROM bank that contains practice hack code, meaning that the results will very likely not be vanilla.
Volatile warning
This icon will display if the buffer is pointing to WRAM banks ($7E and $7F), SRAM banks, WRAM/hardware register mirrors, or open bus. Basically anything that contains volatile memory and is likely to cause a crash.
WARNING:The vanilla code warning can only determine where practice hack code is by the bank pointer (highest byte). There is still potential for non-vanilla memory to be read as data in other portions of ROM.

Rows 3 & 4

These rows contain the same information for different properties. Row 3 contains information about camera control on the X-axis. Its highlight information is in white text. Row 4 contains information about camera control on the Y-axis. Its highlight information is in yellow text.

Camera scroll
The first value on each row displays the current scroll value of the camera on each axis, by reading addresses $E2(2 bytes) for the X-axis and $E8(2 bytes) for the Y-axis.
Camera sync
Each row shows the sync status of the camera on that particular axis. This is determined by whether or not the value is within the current active boundaries.
Room boundaries
The next 2 values in each row display the upper and lower bounds of the camera scroll on that particular axis for the current quadrant. This value is updated based on the current room's layout.
Tile boundaries
The final 2 values in each row display the upper and lower bounds of the camera scroll on that particular axis for the supertile. This value appears to remain static once it loaded for a room.
Boundaries and activeness

Boundaries control the upper and lower limit of camera scroll on each axis. In other words, if the camera approaches the value from above (lower boundaries) or from below (upper boundaries), it will stop scrolling at that value.

Only one of the quadrant and tile boundary sets is active on each axis. This is determined by a number of factors, such as room layout. The set to use is stored at $A6 for the X-axis and $A7 for the Y-axis. When the value is $00, that axis will use the room boundaries. When the value is $02, that axis will use the tile boundaries

The active boundary set for each axis will have its icons and values highlighted in white or yellow, for the X- and Y-axes, respectively. The inactive boundary set for each access will have its icons and values in gray.

Explanation of icons

The actual value is in sync with the expected value, or (for camera) within the expected bounds.
The actual value is out of sync with the expected value, or (for camera) outside the expected bounds.
The value that follows is a correction of the previous, based on certain presumptions.
White (or yellow) text, black outline
The value is read from memory. Yellow text indicates that the value is for the Y-axis. This is solely to distinguish is from the X-axis values more readily.
White text, red outline
Expected value that does not match the actual value
Gray text
Expected value that matches the actual value, and is thus not problematic, or (for camera) the inactive boundaries.
